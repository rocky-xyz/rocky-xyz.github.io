RBAC vs ABAC

1. Requirement Analysis

Static and dynamic permissions

  • Static (RBAC): permissions table: Static definition of operations (e.g., Model: Create)
  • Dynamic (ABAC): policies table: Defines the conditions that must be met when performing these operations (e.g., only allow reading models created by oneself, requiring both model and user attributes), or used to handle dynamically changing attributes (IP, timestamp)

Role inheritance

  • Advanced users automatically inherit all the permissions of junior users and can additionally acquire some other permissions
  • Add a parent_id column in the roles table, sourced from users.id, to establish a hierarchical relationship among roles

Temporary Permissions

  • The user is temporarily granted a role with a validity period of only one month.
  • Add the start_time and end_time fields to the user_roles table.

Multi-team consideration

  • If the algorithm platform serves multiple isolated teams
  • It is necessary to maintain a team table, and all core tables must include a team_id column. If users from TeamA and TeamB create the same entity (e.g., role, policy), the entity must have a clear ownership marker to ensure data isolation

User-Item Many-to-Many Relationship

  • A projects table needs to be maintained, which is associated with the users table
